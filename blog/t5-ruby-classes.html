Shuffle Up and Deal: A Sample Ruby Class

Nov. 9th, 2014

What are Classes?

Everything in Ruby is an object. However, objects are sorted into many different categories called Classes, all of which have different attributes. The most common classes are things like Integers and Floats (rounded vs decimal numbers), Strings (letters and words), and data structures like Hashes and Arrays.

But you aren't limited to just those ubiquitous classes. You can make your own classes in Ruby, and give them methods. Methods are like the "verbs" to Classes "nouns" in Ruby. Why would you want to make your own class?

First off, why wouldn't you? Making things is fun! But in all seriousness, making classes is a helpful way to model real-world objects (there's that word again), which makes it easier to understand how things work (and to explain it to the computer).

Another useful aspect of making classes is that doing so allows you do divide large, unwieldy methods into smaller, bite-sized methods. This is important for a few reasons, like readability and debugging, but also because it just makes sense sometimes.

For example, you could have a program that simulated someone playing football. It might look like this:


              def play_football
              # run the ball
              # throw the ball
              # catch the ball
              # block that kick
              # celebrate after scoring
              end

That's a lot of stuff crammed into one method, right?

But if you break up the methods more discretely using a class, it looks like this:


              class FootballPlayer

              def tackle
              # wrap up offensive player's legs
              # drag offensive player to ground
              end

              def pass
              # throw ball to @variable_representing_receiver
              end

              def celebrate
              # jump in the stands with the fans
              end

              end

I realize that is just some gibberish code, but I hope it illustrates the concept of using a class in order to contain many small methods instead of cramming a ton of actions into one large method.

Anatomy of A Class

The first thing you'll see in a class is the Class name. And yes, it does need to be CamelCase capitalized like that.

class DeckOfCards

The next part is the initialize method. The initialize method dictates what happens when an instance of new class is created. Creating a class is done like this:

deck = DeckOfCards.new

Instance variables (like @deck) are different than normal variables. Marked by an @ symbol, instance methods act as attributes that exist across methods within a class. In this case, the instance variable @deck will be accessible by other methods in the DeckOfCards class.

Okay, so we have a (virtual) deck of cards. What can we do with it? Let's look at some more methods.


              def reveal
              p "Here's the deck."
              puts @deck.values
              end

This method tells the user "Here's the deck" and then prints the values of all of the key/value pairs in the @deck Hash to the console. Note the use of the instance variable @deck. We didn't need to define what @deck was in this method because instance variables can be used in other methods.


              def shuffle
              p "Let's shuffle 'em up."
              @shuffled_deck = @deck.values.shuffle!
              puts @shuffled_deck
              end

This method tells the user "Let's shuffle 'em up.". It then creates an instance variable called @shuffled_deck which is an Array made up of shuffled values from key/value pairings in the @deck Hash, which it then prints. You can see how using instance variables instead of redefining local variables in each method can be helpful for transferring information and reducing redundancy.


              def random_card
              @random_card = @shuffled_deck.shuffle![0]
              p "Here's a random card. It's the #{@random_card}."
              end

This method creates an instance variable called @random_card, which it determines by taking the 0th index (1st card) of the shuffled @shuffled_deck Array (which, if you'll recall, is made up of values from the key/value pairs that were in the @deck Hash).

Classes, For Instance

So that was a lot of information and terminology, perhaps not explained in the easiest way. Let's recap a few of the highlights:

  • Classes are useful for modeling real-world object
  • You can make your own Classes
  • Classes allow you to break up long methods into several methods and keep them all in one place
  • Instance variables are cool because they can be used in different methods within a class instead of having to be redefined like local variables
  • New instances of a class are initialized with attributes determined by the initialize method

I mention inheritance or attribute reader/writer methods, but trust me, those are awesome too. I'll just say a little here. Inheritance basically lets you have a class inherit the traits of a class above it. This works similarly to genetics and biological classification (humans love categorizing things, don't we?). Attribute reader and writer methods are a way of saving time/code. Simply put, this: 


              def age
              @age = age
              end

              def change_age(new_age)
              @age = new_age
              end

is equivalent to this:


              attr_reader :age
              attr_writer :age
