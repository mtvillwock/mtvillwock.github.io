JavaScript Constructor Functions versus Ruby Classes

Nov. 23rd, 2014

And then there was JavaScript

This week DBC pulled a trick on me. Well, on all of us boots. What trick is that?

JavaScript.

Now, I'm not going to rant about JavaScript too much because I actually don't think it is that bad. I want to keep myself in a growth-mindset (and this week reinforced how important that was for me). The short version is that I had some struggles during my transition from Ruby to JavaScript. But enough about me, let's get to the languages.

What are the differences between these two object-oriented languages? There are many. But let's start with one that gave me trouble (until I read this): Ruby Classes versus JavaScript constructor functions.

Ruby Classes

You've seen these bad boys before. Classes are used to model real-life objects, like a person or a car or a tool. For example:


class FootballPlayer
  def initialize(name)
    @name = name;
    @has_ball = false;
    @down = false;
    @x = 1
    @y = 5
  end

  def status
  # Convoluted method here
  end

  def move_right
    if @down == true
    puts "#{@name} is down. The game is over."
    else
    @x += 1
    puts status
    end
  end

end

packer = FootballPlayer.new("Micah Hyde")
packer.status
packer.move_right

The code above would return this when run:


#=> Micah Hyde is at the 10 yard line.
#=> Micah Hyde is at the 20 yard line.

Pretty simple, right? You make a new instance of the class FootballPlayer, which is initialized with a name that you provide, and some important variables like his location (x and y) and whether or not he has the ball or is down. It also gives him two methods: status (which I didn't include here but is a long method that checks where is he and then gives a response based on many different possible conditions) and move_right, which moves the player right and then calls the aforementioned status method. But what does that look like in JavaScript?

JavaScript constructor functions

The thing with JavaScript is that it is not a pretty language. Ruby does everyone a favor by clearly delineating things in a relatively English-like manner. Not so, JavaScript. Here is the same "class" from above written as a JavaScript constructor function:


function football_player (name) {
  this.name = name;
  this.hasBall = false;
  this.down = false;

  this.status = function() {
  // insufferable amounts of conditional statements here
}

  this.moveRight = function() {
    if (this.down === true) {
    console.log(this.name +" is down. The game is over.");
  }
    else {
    this.x +=1; this.status();
  }
  }
}

packer = new football_player("Micah Hyde")
packer.status();
packer.moveUp();

#=> Micah Hyde is at the 10 yard line.
#=> Micah Hyde is at the 20 yard line.

Like before, this gives us an object (the football player) with the properties name, hasBall, and down as well as the function moveRight. These are respectively equivalent to the instance variables in the def initialize method and the def move_right method in the Ruby class above. As you can see, the outputs are similar.

When you get down to it, both snippets of code are doing the same thing. The difference is in the syntax and in some of the rules of what interacts with what. In Ruby you define the scope of your variables by given them a prefix like $, @, or @@. This allows the variables to be used beyond the local scope of whatever code block or method they are being used in. But in JavaScript the scope of your variables (and functions) is based on where they are inside of a function. This is what tripped me up. In Ruby you can initialize an instance variable and use it throughout all the other methods in a class. In JavaScript the scope of your variable depends on where you put it inside of a large function.

I suppose in hindsight, JavaScript makes sense with its variables, insofar as having their locality based on how deep or shallowly they are located inside of a large function or method (a method in JavaScript meaning a function, such as moveRight above, inside of a larger function that represents what would be a class in Ruby). I like the way the scope of Ruby's variables are distinguished by special symbols and not by where they are written. But JavaScript is an essential tool for working with web browsers, and as such I will not let a little thing like syntax keep me from adding it to my toolbox.

I hope this post made sense and taught you a bit about one difference between Ruby and JavaScript. Thanks for reading!
